# 租户基座（Tenancy Foundation）设计

- 日期：2026-06-08
- 分支：`feature/multi-business`
- 状态：待评审（已并入 codex 评审 16 项 + 3 个决策）
- 作者：李碧强 + Claude

## 这是什么

把当前**单租户** SaaS（一切挂在 `User` 名下）改造成**企业多租户**架构的地基。本子项目只做「身份与隔离的基座」，不做品牌、定价、结算等上层运营功能——那些是后续子项目，依赖本基座。

本基座做完后，系统具备：每条业务数据归属某个租户、按域名解析当前租户、查询层自动隔离、**认证/异步入口/实时通道都绑定租户**、存量数据安全迁移到「主站」。

## 背景与约束（已确认的决策）

- **业务形态**：企业多租户，按组织隔离。
- **入口**：独立域名/子域名 + 品牌主题（真正「分站点」）。用户从哪个站进来就归哪个租户。
- **账号归属**：租户隔离账号。同一手机号可在不同站各自注册、互不干扰。
- **第三方身份（决策 codex#4）**：微信 `openId/unionId`、Watcha 改为**租户内唯一**，同一微信号可在不同分站各自绑定独立账号——与手机号一致。
- **未知域名（决策 codex#7）**：生产环境收到未在 `TenantDomain` 登记的 Host **一律拒绝（404/「站点未配置」页）**，不兜底主站。主站域名也必须显式登记。
- **DB 层兜底（决策 codex#15）**：基座阶段用**轻量方案**——Prisma 扩展主防 + CI 阻断裸 SQL + `PrismaService` 封装审计。Postgres RLS 留作后续加固选项。
- **收款**：统一进平台账户，平台再跟租户结算（本基座不实现结算，但保证 AI 用量/订单可按租户归因）。
- **AI 货源**：平台共享 new-api 池；按租户分配「中转 key」做用量统计是子项目 4。
- **唯一真值**：`Tenant.id`，**不可变**。域名是**可变映射**，一租户可多域名。
- **数据行主站语义**：业务行 `tenantId` 为空或 `'default'` 都归主站（数据层）。**注意：这与入口层「未知 Host 拒绝」是两层，不可混淆**——数据行兜底 default 安全，入口 Host 兜底 default 是漏洞。
- **隔离强制**：CLS（AsyncLocalStorage）+ Prisma 查询扩展**自动注入** `tenantId`，业务 service 不手写。

## 架构

四层，从入口到数据，外加横切的「异步入口」与「实时通道」：

```
① 入口解析层（HTTP）
   Host 头 → 规范化（lowercase/去端口/punycode/去尾点）
          → 查 TenantDomain.host → tenantId
          → 命中：该租户；未命中：生产拒绝 404（不兜底主站）
          → cls.run({ tenantId, slug, isPlatform }) 包裹后续处理

①' 认证层（codex#2/#3）
   JWT payload 含 tenantId；JwtStrategy 校验 payload.tenantId === 当前Host租户
   不一致 → 401。RefreshToken 带租户维度。

①'' 异步入口层（codex#1/#9）—— 无 Host，禁止默认主站
   支付回调 / cron / worker / MQ：
     先以平台态按业务行（orderNo/taskId/userId）查出 tenantId
     → runAsTenant(tenantId) 包裹处理逻辑
     绝不依赖「CLS 空 = default」兜底来发货/扣费

① ''' 实时通道层（codex#13）
   WS/SSE 连接握手时：从 Host + JWT 解析 tenantId 并绑定到连接
   room key 加租户命名空间：`t:<tenantId>:team:<teamId>` 等

② 隔离强制层
   PrismaService 加 $extends 查询扩展，从 CLS 读 tenantId：
      find*/update*/delete*/count/aggregate/groupBy → 注入 where:{ tenantId }
      create/createMany                            → 注入 data:{ tenantId }
      upsert                                       → where + create + update 三处都注入
      全局白名单 model                              → 跳过
      runAsPlatform()（仅 platform-admin）          → 临时关闭注入
   ⚠ 局限：扩展只保证「行本身的 tenantId」，不校验 relation/connect 的外键所属租户
     → 写入前业务层校验外键同租户（见「外键跨租户校验」）

③ 数据层
   业务表含 tenantId（NOT NULL DEFAULT 'default'）
   全局白名单表不含 tenantId（含定价/套餐表，基座阶段）
   逐表定义唯一键的租户语义（见「唯一键策略」）
```

### 组件清单

| 组件 | 职责 |
|------|------|
| `Tenant` / `TenantDomain` 模型 | 租户真值 + 域名映射 |
| `TenantContext`（CLS service） | 请求级存取 `tenantId`；`runAsTenant(id)` / `runAsPlatform()` |
| `TenantResolverMiddleware` | Host 规范化 → tenantId → CLS；未知 Host 拒绝 |
| `JwtStrategy` 增强 | 校验 token 租户 === Host 租户 |
| Prisma 租户扩展 | 自动注入/过滤；upsert 全注入；白名单跳过 |
| 异步入口包装 | 支付回调/cron/worker 用 `runAsTenant(row.tenantId)` |
| 实时通道绑定 | WS/SSE 连接绑 tenantId + room key 命名空间 |
| 外键同租户校验 | 写入前校验引用的 user/team/project 同租户 |
| 迁移脚本（分阶段） | 建主站 + 回填 + 改约束 + 并发建索引 |
| CI 裸 SQL 闸 | 阻断/审计 `$queryRaw`/`$executeRaw` |

## 数据模型

### 新增

```prisma
model Tenant {
  id          String         @id @default(uuid())   // 唯一真值，永不变；主站固定为 'default'
  slug        String         @unique                // 人类可读短码，如 "acme"，可改
  name        String
  status      String         @default("active")     // active | suspended
  isPlatform  Boolean        @default(false)        // 主站标记
  createdAt   DateTime       @default(now())
  updatedAt   DateTime       @updatedAt
  domains     TenantDomain[]

  @@index([status])
}

model TenantDomain {
  id        String   @id @default(uuid())
  tenantId  String
  host      String   @unique                        // 已规范化：lowercase、无端口、punycode
  isPrimary Boolean  @default(false)                // 主域名（生成回跳链接用）
  verified  Boolean  @default(false)                // 自定义域名归属校验（子项目 2 用）
  createdAt DateTime @default(now())
  tenant    Tenant   @relation(fields: [tenantId], references: [id], onDelete: Cascade)

  @@index([tenantId])
}
```

### `User` 改动（账号 + 第三方身份租户化）

```prisma
model User {
  // ...
  tenantId             String  @default("default")
  // 删除以下单列 @unique：phone / email / watchaUserId / wechatOfficialOpenId / wechatUnionId
  @@unique([tenantId, phone])
  @@unique([tenantId, email])
  @@unique([tenantId, watchaUserId])           // codex#4：租户内唯一
  @@unique([tenantId, wechatOfficialOpenId])
  @@unique([tenantId, wechatUnionId])
  @@index([tenantId, phone, status])
  @@index([tenantId, wechatUnionId])
}
```

> **配套改 auth 查找点（codex#3，上线阻断项）**：当前 `users.service` 等处大量 `findUnique({ where: { phone } })` / `{ email }` / 微信 openId 查找，迁移后单列 unique 不存在，这些调用**编译即失败**。spec 实现阶段必须枚举并重写为 `findFirst({ where: { tenantId, phone } })` 或复合 unique input `findUnique({ where: { tenantId_phone: { tenantId, phone } } })`。受影响点（实现时逐一核对）：
> - 手机号登录/注册/改密
> - 邮箱登录/找回
> - 微信扫码登录与绑定（`WechatLoginSession` → openId/unionId 查 User）
> - Watcha 登录绑定
> 微信扫码登录的租户来源 = 发起扫码的站点（`WechatLoginSession` 增 `tenantId`，回调按它定位）。

### 加 `tenantId` 的业务表

`User`、`Project`、`WorkflowHistory`、`UserTemplate`、`GlobalImageHistory`、
`CreditAccount`、`CreditTransaction`、`CreditLot`、`CreditConsumePolicy`、`CreditAnomalyRecord`、`ApiUsageRecord`、
`UserMembershipSubscription`、`MembershipSubscriptionChange`、`MembershipEntitlementSnapshot`、
`InvitationCode`、`InvitationRedemption`、
`PaymentOrder`、`ImageTask`、`VideoTask`、
`WechatLoginSession`、`RefreshToken`、
`Team`、`TeamMembership`、`TeamInvite`、`TeamProjectShare`、`TeamCreditAccount`、`TeamCreditLot`、`TeamCreditLedger`、`TeamSeatPackage`、`TeamSubscriptionPlan`、`TeamSubscription`、
`VolcReviewGroup`、`BioAuthGroup`

> `ApiUsageRecord/ImageTask/VideoTask` 必须落 `tenantId`（codex#11）——这是子项目 4「分 key 统计」准确回填的前提，基座就要保证 AI 用量可按租户归因。

### 全局白名单表（**不**加 `tenantId`，扩展跳过注入）

`Tenant`、`TenantDomain`、`SystemSetting`、`NodeConfig`、`PublicTemplate`、`postgres_log`、健康检查/基础设施表，
**外加（codex#10 决策）**：`CreditPricing`、`CreditPackage`、`MembershipPlan`、`CreditConsumePolicy`（全局计费策略，全平台共享一套）。

> 这三张定价/套餐表基座阶段保持全局，**所有租户都读主站同一套定价**——避免「加了 tenantId 默认 default、扩展又注入当前租户 → 非主站租户查不到套餐 → 会员购买/计费直接坏」。真正的租户级定价（含「租户无配置则继承主站」的继承模型）留到子项目 5。

> 白名单是一份**显式常量列表**，扩展据此跳过。新增表默认归「租户表」（更安全），要进白名单必须显式加入并评审。

### 唯一键的租户语义（codex#5，逐表策略）

| 表 / 字段 | 现状 | 基座策略 |
|-----------|------|----------|
| `User.phone/email` | 全局唯一 | **租户内唯一** `(tenantId, x)` |
| `User.watchaUserId/wechat*` | 全局唯一 | **租户内唯一** `(tenantId, x)`（决策 codex#4） |
| `CreditAccount.userId @unique` | 全局唯一 | **租户内唯一** `(tenantId, userId)`（userId 已租户内唯一，等价保持） |
| `MembershipEntitlementSnapshot.userId @id` | 主键=userId | 加 tenantId 列；userId 仍租户内唯一，主键不变 |
| `MembershipPlan.code @unique` | 全局唯一 | **白名单表，保持全局唯一**（基座阶段全平台共享） |
| `CreditConsumePolicy.code @unique` | 全局唯一 | **白名单表，保持全局唯一**（全平台共享一套计费策略） |
| `InvitationCode.code` / `TeamInvite.code @unique` | 全局唯一 | **保持全局唯一**——邀请码是不可猜 token，跨租户也不能撞，全局唯一更安全 |
| `PaymentOrder.orderNo @unique` | 全局唯一 | **保持全局唯一**——回调按 orderNo 平台态定位租户（见下） |

> 原则：**用户可见的业务键** → 租户内唯一；**不可猜的安全 token / 全局票据**（邀请码、订单号）→ 保持全局唯一并配合平台态定位。

## 数据流

### 普通 HTTP 请求（自动隔离）

```
1. Host=acme.tanva.com → 规范化 → 查 TenantDomain → tenantId='t_acme'
   （未登记域名 → 404「站点未配置」，不进业务）
2. JwtStrategy：校验 token.tenantId === 't_acme'，否则 401
3. cls.run({ tenantId:'t_acme' }, () => handler())
4. service 调 prisma.project.findMany({ where:{ userId } })
   → 扩展注入 → { userId, tenantId:'t_acme' }
5. 仅返回该租户数据
```

### 支付回调 / 异步入口（codex#1/#9，禁止默认主站）

```
微信/支付宝回调（无 Host，仅 orderNo）：
1. runAsPlatform(() => prisma.paymentOrder.findUnique({ where:{ orderNo } }))
      → 拿到 order.tenantId
2. runAsTenant(order.tenantId, async () => {
      processPaymentSuccess(order)   // 发积分/会员/团队席位，全在正确租户上下文
   })
cron / worker 同理：按业务行的 tenantId 切上下文，或对每个活跃租户循环 runAsTenant。
```

### 主站超管跨租户（逃生舱 + 权限边界，codex#12）

```
runAsPlatform(async () => { ... })   // 闭包内不注入 tenantId，可跨租户
权限：仅 platform-admin（独立于租户内 admin 角色）可调用；
     且 platform-admin 仅在主站登录态（token.tenantId==='default' 且 user 为平台超管）生效。
     租户内 admin 永远拿不到 runAsPlatform。
```

### 外键跨租户校验（codex#6）

扩展只保证「写入行的 tenantId」，不校验 `connect`/外键引用是否同租户。约定：
- 创建/更新涉及跨实体引用（如 `TeamProjectShare(projectId, teamId)`、支付按 `metadata.teamId` 建 `TeamSeatPackage`）时，**业务层先校验被引用实体 `tenantId === 当前租户`**，再写入。
- 关键关系评估加**复合外键** `(tenantId, id)` 以让数据库兜底（实现时按收益逐个加）。

## 错误与边界处理

- **未知 Host**：生产 404「站点未配置」（决策 codex#7）。开发环境可放宽到 default，由 env 开关控制。
- **Host 规范化（codex#8）**：lowercase、去端口、去尾点、punycode；`x-forwarded-host` 仅信任已知反代（白名单 IP/固定头），否则用 `host`。规避同域多记录/绕过验证。
- **CLS 未初始化**：在「写路径」视为错误而非 default——异步任务必须显式 `runAsTenant`/`runAsPlatform`，否则抛错，避免静默落错租户。（仅纯读的无害场景可回落 default。）
- **token 与 Host 租户不一致**：401（codex#2），防跨站 cookie 串用户。
- **裸 SQL 绕过扩展（codex#15）**：CI 扫描 `$queryRaw`/`$executeRaw` → 默认阻断，例外需显式标注并带 `tenant_id`；`PrismaService` 封装 raw 入口集中审计。
- **逃生舱滥用**：`runAsPlatform` 只在平台超管模块导出/调用，普通业务模块不引入。

## 测试策略（codex#16，覆盖事故面）

**隔离正确性**
- 两租户各建数据，断言 A 查询拿不到 B（find/count/update/delete/aggregate 各一）。
- 不传 tenantId 创建 → 落 CLS 当前租户。
- 白名单表（含 `MembershipPlan`）查询**不**被注入 tenantId，所有租户读到同一套定价。
- `upsert`/`connect`/`include` 隔离：断言无法借 relation 跨租户读写。

**认证与入口**
- 跨租户 JWT：A 站 token 请求 B 站 → 401。
- 未知 Host 生产 → 404。
- 微信/Watcha 同一身份在两站各自注册/绑定 → 两个独立账号（codex#4）。

**异步入口（最易出事）**
- 支付回调对**非 default 租户**订单正确发货（积分/会员/团队席位）。
- 补单/过期清理 cron 对非 default 租户生效。
- worker 处理非 default 租户 AI 任务，`ImageTask/VideoTask` 落正确 tenantId。

**外键边界**
- Team+Project 跨租户引用（A 租户 project + B 租户 team）→ 拒绝。

**防御纵深**
- raw SQL CI 扫描：新增未带 tenant_id 的 `$queryRaw` → CI 失败。

**迁移**
- 存量库快照跑迁移 → `count(tenantId IS NULL)=0`；主站登录/出图/列表回归冒烟。

## 迁移步骤（分阶段，规避 PG 锁，codex#14）

```
阶段 A — 元数据
  1. 建主站：Tenant{ id:'default', slug:'platform', name:'主站', isPlatform:true }
     TenantDomain{ host:<规范化主站域名>, tenantId:'default', isPrimary:true, verified:true }

阶段 B — 加列（不锁全表）
  2. 各业务表加 tenantId 列：nullable + DEFAULT 'default'（PG 常量 default 不重写全表）
  3. 批量回填存量行 tenantId='default'（分批 update，避免长事务）

阶段 C — 约束（避免长锁）
  4. 加 CHECK(tenantId IS NOT NULL) NOT VALID → VALIDATE CONSTRAINT（分两步，校验不持写锁）
  5. 或在确认回填完成后 SET NOT NULL（评估表大小，必要时用 NOT VALID check 替代）

阶段 D — 唯一约束与索引（并发，避免阻塞写）
  6. 前置校验：当前无会冲突数据（全归主站，等价旧全局唯一）
  7. CREATE UNIQUE INDEX CONCURRENTLY 新复合唯一（tenantId, phone/email/wechat*…）
  8. drop 旧单列 unique
  9. 高频 @@index([userId,...]) → CREATE INDEX CONCURRENTLY 升级为 (tenantId, userId, ...)

阶段 E — 校验
  10. 所有租户表 count(tenantId IS NULL)=0；主站功能回归冒烟
```

**回滚注意（codex#14）**：不是简单 drop 列——一旦非主站租户产生了跨租户重复手机号/email，drop 复合唯一恢复全局唯一会**直接失败**。回滚预案：先确认未产生跨租户重复，再还原；否则需先归并/清理重复账号。基座上线初期（仅主站一个租户）回滚是安全的，多租户开站后回滚需谨慎。

## 明确不做（YAGNI / 留给后续子项目）

- 品牌主题、登录页定制、自定义域名归属校验流程 → 子项目 2
- 租户管理后台、主站超管控制台 UI → 子项目 3
- 按租户分 new-api 中转 key、用量统计与结算回填 → 子项目 4
- 租户级定价/会员/积分池下放 + 继承模型 → 子项目 5
- 收款与平台↔租户结算 → 子项目 6
- Postgres RLS 行级安全加固 → 后续可选（基座用轻量 CI 闸）
- 本基座**不**改业务逻辑本身，只加隔离维度 + 认证/异步/实时绑定 + 迁移。

## 验收标准

- [ ] `Tenant` / `TenantDomain` 就绪，主站 `'default'` + 已登记主站域名存在。
- [ ] 业务表含 `tenantId NOT NULL DEFAULT 'default'`；白名单（含定价/套餐三表）不含。
- [ ] `User` 及第三方身份唯一约束为 `(tenantId, x)`；邀请码/订单号保持全局唯一。
- [ ] 所有 auth 查找点重写为复合 unique / `findFirst({tenantId,...})`，编译通过。
- [ ] Host 规范化 + 未知 Host 生产拒绝；JWT 校验租户一致。
- [ ] Prisma 扩展自动注入/过滤（含 upsert），白名单跳过，`runAsPlatform`/`runAsTenant` 可用。
- [ ] 支付回调/cron/worker 对非 default 租户正确处理（不依赖 default 兜底）。
- [ ] WS/SSE 绑定 tenantId + room key 命名空间。
