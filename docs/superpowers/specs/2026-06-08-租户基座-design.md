# 租户基座（Tenancy Foundation）设计

- 日期：2026-06-08
- 分支：`feature/multi-business`
- 状态：待评审
- 作者：李碧强 + Claude

## 这是什么

把当前**单租户** SaaS（一切挂在 `User` 名下）改造成**企业多租户**架构的地基。本子项目只做「身份与隔离的基座」，不做品牌、定价、结算等上层运营功能——那些是后续子项目，依赖本基座。

本基座做完后，系统具备：每条业务数据都归属某个租户、按域名解析当前租户、查询层自动隔离、存量数据安全迁移到「主站」。

## 背景与约束（已确认的决策）

- **业务形态**：企业多租户，按组织隔离。
- **入口**：独立域名/子域名 + 品牌主题（真正「分站点」）。用户从哪个站进来就归哪个租户。
- **账号归属**：租户隔离账号。同一手机号可在不同站各自注册、互不干扰。
- **收款**：统一进平台账户，平台再跟租户结算（本基座不实现结算，但隔离要支持后续按租户统计）。
- **AI 货源**：平台共享 new-api 池；后续按租户分配「中转 key」做用量统计（子项目 4，不在本基座）。
- **唯一真值**：`Tenant.id`，**不可变**。域名是**可变映射**，一租户可多域名。
- **主站语义**：`tenantId` 为空 **或** 等于 `'default'` 都算主站。存量无 `tenantId` 的数据归主站。
- **隔离强制**：CLS（AsyncLocalStorage）+ Prisma 查询扩展**自动注入** `tenantId`，业务 service 不手写。

## 架构

三层，从入口到数据：

```
① 入口解析层
   HTTP Host 头 → TenantResolverMiddleware
      查 TenantDomain.host → tenantId
      命中 → 该租户；未命中 → 主站 'default'（兜底）
      把 { tenantId, slug, isPlatform } 存入 CLS（AsyncLocalStorage）

② 隔离强制层
   PrismaService 加 $extends 查询扩展，从 CLS 读 tenantId：
      find*/update*/delete*/count → 自动注入 where: { tenantId }
      create/createMany           → 自动注入 data:  { tenantId }
      全局白名单 model            → 跳过，不注入
      逃生舱 runAsPlatform()       → 临时关闭注入（仅主站超管接口可用）

③ 数据层
   业务表全部含 tenantId 列，NOT NULL DEFAULT 'default'
   全局白名单表不含 tenantId
   Tenant.id='default' 为主站；User 唯一约束改 (tenantId, phone)
```

### 组件清单

| 组件 | 职责 | 依赖 |
|------|------|------|
| `Tenant` / `TenantDomain` 模型 | 租户真值 + 域名映射 | Prisma |
| `TenantContext`（CLS service） | 请求级存取当前 `tenantId` | nestjs-cls / AsyncLocalStorage |
| `TenantResolverMiddleware` | Host → tenantId，写入 CLS | TenantDomain 查询 + 缓存 |
| Prisma 租户扩展 | 自动注入/过滤 `tenantId` | TenantContext + 全局白名单 |
| `runAsPlatform()` 逃生舱 | 跨租户/超管场景临时绕过 | TenantContext |
| 迁移脚本 | 建主站 + 回填 + 改约束 | Prisma migrate |

## 数据模型

### 新增

```prisma
model Tenant {
  id          String         @id @default(uuid())   // 唯一真值，永不变；主站固定为 'default'
  slug        String         @unique                // 人类可读短码，如 "acme"，可改
  name        String
  status      String         @default("active")     // active | suspended
  isPlatform  Boolean        @default(false)        // 主站标记
  createdAt   DateTime       @default(now())
  updatedAt   DateTime       @updatedAt
  domains     TenantDomain[]

  @@index([status])
}

model TenantDomain {
  id        String  @id @default(uuid())
  tenantId  String
  host      String  @unique                         // "acme.tanva.com" / "design.acme.com"
  isPrimary Boolean @default(false)                 // 主域名（生成回跳链接用）
  verified  Boolean @default(false)                 // 自定义域名归属校验（子项目 2 用）
  createdAt DateTime @default(now())
  tenant    Tenant  @relation(fields: [tenantId], references: [id], onDelete: Cascade)

  @@index([tenantId])
}
```

### 改动现有表

**所有业务表**加：

```prisma
tenantId String @default("default")   // NOT NULL DEFAULT 'default'
@@index([tenantId])                    // 单列；高频列表查询再加复合索引
```

**`User`** 额外改唯一约束：

```prisma
tenantId String @default("default")
// 删除 phone @unique / email @unique 的单列唯一
@@unique([tenantId, phone])
@@unique([tenantId, email])
@@index([tenantId, phone, status])
```

> 复合索引收益：现有 `@@index([userId, ...])` 这类列表查询，配合自动注入的 `tenantId` 过滤，建议升级为 `@@index([tenantId, userId, ...])`，避免隔离后走不到索引。实现时逐表评估，不一刀切。

### 加 `tenantId` 的业务表（清单）

`User`、`Project`、`WorkflowHistory`、`UserTemplate`、`GlobalImageHistory`、
`CreditAccount`、`CreditTransaction`、`CreditLot`、`CreditConsumePolicy`、`CreditAnomalyRecord`、`ApiUsageRecord`、
`CreditPricing`、`CreditPackage`、`MembershipPlan`、`UserMembershipSubscription`、`MembershipSubscriptionChange`、`MembershipEntitlementSnapshot`、
`InvitationCode`、`InvitationRedemption`、
`PaymentOrder`、`ImageTask`、`VideoTask`、
`WechatLoginSession`、`RefreshToken`、
`Team`、`TeamMembership`、`TeamInvite`、`TeamProjectShare`、`TeamCreditAccount`、`TeamCreditLot`、`TeamCreditLedger`、`TeamSeatPackage`、`TeamSubscriptionPlan`、`TeamSubscription`、
`VolcReviewGroup`、`BioAuthGroup`

> `CreditPricing`/`CreditPackage`/`MembershipPlan` 本基座只加列默认 `'default'`，真正下放到子项目 5。

### 全局白名单表（**不**加 `tenantId`）

`Tenant`、`TenantDomain`、`SystemSetting`、`NodeConfig`、`PublicTemplate`、`postgres_log`，以及健康检查/基础设施表。

> 白名单是一份**显式常量列表**，Prisma 扩展据此跳过注入。新增表时必须明确归类（默认归「租户表」，更安全）。

## 数据流

### 普通请求（自动隔离）

```
1. 请求到达，Host = acme.tanva.com
2. TenantResolverMiddleware：
     查缓存 → 未命中查 TenantDomain.host='acme.tanva.com' → tenantId='t_acme'
     未匹配到任何域名 → tenantId='default'（主站兜底）
     cls.set('tenantId', tenantId)
3. Controller/Service 调 prisma.project.findMany({ where: { userId } })
4. Prisma 扩展介入 → where 变成 { userId, tenantId: 't_acme' }
5. 只返回该租户数据
```

### 写入

```
prisma.project.create({ data: { userId, name } })
   → 扩展注入 → data: { userId, name, tenantId: 't_acme' }
```

### 主站超管跨租户（逃生舱）

```
runAsPlatform(async () => {
   // 此闭包内扩展不注入 tenantId，可查所有租户
   return prisma.tenant.findMany() // 或带显式 where: { tenantId } 精确查某租户
})
```

仅平台超管 guard 通过的接口可调用 `runAsPlatform`。

## 错误与边界处理

- **域名未匹配**：兜底主站 `'default'`，不报错（与「空也算主站」一致）。
- **CLS 未初始化**（如启动期定时任务、脚本）：扩展读到空 → 视为 `'default'`。需要跨租户的后台任务**显式**用 `runAsPlatform` 或对每个租户循环 `runAsTenant(id)`。
- **裸 SQL（`$queryRaw`/`$executeRaw`）绕过扩展**：
  - 一次性审计现有所有裸 SQL，手动补 `tenant_id` 条件；
  - 约定：以后裸 SQL **必须**显式带 `tenant_id`；
  - 在 PR 模板/评审清单加一条提醒。
- **User 唯一约束迁移失败**（存量是否已有跨租户重复手机号？）：迁移前所有数据都属主站，`(tenantId, phone)` 等价于旧的 `phone @unique`，不会冲突；迁移脚本仍加一步前置校验。
- **逃生舱滥用**：`runAsPlatform` 只在主站超管模块导出/调用，普通业务模块不引入；评审把关。

## 测试策略

- **隔离单测**：构造两个租户各建数据，断言租户 A 的查询拿不到 B 的行（find/count/update/delete 各一）。
- **写入注入测试**：不显式传 `tenantId` 创建，断言落库的是 CLS 当前租户。
- **白名单测试**：对 `SystemSetting` 等白名单表的查询断言**不**被注入 `tenantId`。
- **兜底测试**：未知域名 / 无 CLS → 解析为 `'default'`。
- **逃生舱测试**：`runAsPlatform` 内可跨租户查询；闭包结束后自动恢复隔离。
- **迁移回归**：在存量库快照上跑迁移，断言 `count(tenantId IS NULL)=0`，主站登录 + 出图冒烟通过。
- **裸 SQL 审计**：grep `$queryRaw`/`$executeRaw`，逐个确认已带 `tenant_id`。

## 迁移步骤（一次性）

```
1. 写入主站租户与域名：
     Tenant     { id:'default', slug:'platform', name:'主站', isPlatform:true }
     TenantDomain { host:<当前主站域名>, tenantId:'default', isPrimary:true, verified:true }
2. 给所有「租户表」加 tenantId 列，NOT NULL DEFAULT 'default'（存量自动回填）。
3. 重建 User 唯一约束：
     前置校验：当前不存在会冲突的数据（全归主站，等价旧约束）
     drop phone @unique / email @unique
     create @@unique([tenantId, phone]) / @@unique([tenantId, email])
4. 按需把高频 @@index([userId, ...]) 升级为 @@index([tenantId, userId, ...])。
5. 校验：所有租户表 count(tenantId IS NULL)=0；主站登录 / 出图 / 列表回归冒烟。
```

迁移可逆性：列带默认值、约束变更有前置校验，回滚即 drop 列 + 还原 `phone @unique`（需先确认未产生跨租户重复）。
