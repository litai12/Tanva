## 1. 两客户端实时协作演示（核心，6/25 主戏）
准备两个浏览器（两个独立 Chrome profile / 一个正常一个无痕，确保 localStorage 隔离），各自登录团队 TAI 的两名成员账号，打开同一个团队项目 `/app`。

逐条对照体验说明演示：
1. **在线**：右上角在线头像条显示 2 人（A、B 头像）。
2. **节点移动 300ms 跟随**：A 拖动一个 Flow 节点 → B 屏幕 <300ms 看到位置变化（实测传输 237ms）。
3. **选中=锁+边框**：A 选中某节点 → B 端该节点出现彩色虚线边框；B 尝试拖动该节点无效（被锁阻止）。
4. **并行编辑**：B 同时编辑/移动另一个节点 → 互不影响。
5. **Prompt 同步**：A 编辑某 Prompt 节点文本 → B 端该节点文本同步更新。
6. **画布图片同步**：A 移动/缩放/删除/插入一张画布图片 → B 端同步（笔迹/涂鸦不实时，属规格"不做"）。
7. **AI 任务状态**：A 发起一次 AI 生成 → B 端看到任务 toast/进度，成功结果落到节点。
8. **保存不互相覆盖**：A、B 各改各的，自动保存互不覆盖（服务端 baseVersion 落后即 409，客户端对齐重试）。
9. **断线重连补帧**：B 断网几秒再恢复 → 自动补齐断线期间 A 的改动；缺帧过多时提示并整页拉全量。

> 自动化证据（已在 2026-06-17 跑过，可复现）：节点移动 414→914 在 A 浏览器实时生效；在线 1→2；锁虚线边框 `outline dashed`；Prompt 文本同步；task_status toast；baseVersion 409；重连 seq 8,9 补齐；canvas_patch 到达浏览器并触发图片插入。复现脚本法：签 JWT(JWT_ACCESS_SECRET) 注入 localStorage(access_token/tanva_active_team_id/current_project_id)，Node ws 充当第二端，task_status 经 Redis PUBLISH `canvas:project:<id>` 注入。

## 2. 团队身份 / 积分语义 演示（C/D + 设计建议 1-7）
- 切到团队：右上角出现「团队名 + 团队」标签、团队色（teal）+ Users 图标的积分 pill（区别于个人金星）。
- 团队模式积分数字 = 个人可用配额；配额不限时显示团队余额 + 「不限」徽章；hover 提示「当前使用团队额度，显示为你的个人可用配额」。
- 点击积分入口 title=「团队额度」，打开团队套餐；团队下拉含「成员管理 / 配额设置」「套餐与账单」。

## 3. 团队积分独立充值（F）——需人工/支付环境验收
- 入口：团队管理弹窗「积分充值」页签（仅 owner/admin 可见）。
- 后端：`POST /api/teams/:teamId/credits/topup-orders` → 创建 `team_credits` 订单（只发积分、不加席位）；支付成功回调 `processPaymentSuccess` 的 `team_credits` 分支给团队账户加积分 + TeamCreditLedger topup + 广播。
- **需人工**：该下单接口受「可信 IP 白名单」保护（本地脚本调用返回 400 是预期的安全控制），且需真实支付网关 + 扫码支付才能端到端验收。请在受信环境/前端发起一次真实小额下单 → 扫码 → 确认团队余额增加且席位数不变。

## 4. 团队积分扣费一致性（#7）——需真实 AI 任务验收
- 该路径用既有 `team-credit-ledger` 的 reserve→deduct/release，本次协作改造未触碰。
- **需人工**：在团队项目里跑一次真实 AI 生成，确认：扣的是团队额度、任务结果正确落画布/Flow、刷新后项目内容与积分账本一致。
